Working notes for LLM assistants. Read this before touching any code.

## Build environment

- **OS:** Debian 11 (Bullseye) in WSL2 on Windows
- **glibc:** 2.31 — prebuilt binaries requiring 2.32+ will not run
- **Compiler:** Clang 11 (`/usr/bin/clang++`)
- **CMake:** 4.3 (installed via pip — the system apt version was 3.18, too old for JUCE 8)
- **Build system:** Ninja
- **JUCE:** 8.0.4 via FetchContent (cached in `build/_deps/` after first configure)

To build:
```bash
cmake -B build -DCMAKE_BUILD_TYPE=Debug -DCMAKE_C_COMPILER=clang \
      -DCMAKE_CXX_COMPILER=clang++ -DCMAKE_EXPORT_COMPILE_COMMANDS=ON -G Ninja
cmake --build build -j$(nproc)
```

VST3 lands at `build/RC20Clone_artefacts/Debug/VST3/RC-20 Clone.vst3`.

## VS Code + WSL setup

VS Code must connect to WSL via the **Remote - WSL** extension (publisher: Microsoft),
not by opening the WSL filesystem path (`\\wsl$\...`) from a Windows VS Code window.
Opening via the network share gives VS Code no access to WSL tools and clangd will
report clang as missing.

**Correct workflow:** from the WSL terminal:
```bash
cd ~/dev/rc-20-clone
code .
```

The bottom-left status bar should read `WSL: Ubuntu` (or your distro name).
The integrated terminal will be a WSL bash/zsh shell with `/usr/bin/clang++` on PATH.

clangd uses `build/compile_commands.json` (generated by `CMAKE_EXPORT_COMPILE_COMMANDS=ON`)
for IntelliSense — no manual compiler path setting needed.

## CI/CD — GitHub Actions

Workflow file: `.github/workflows/build.yml`

- **Every push to `main` / PR:** builds Windows VST3 and macOS VST3 + AU; uploads
  artifacts to the Actions run.
- **Every `v*` tag:** additionally attaches zipped artifacts to the GitHub Release.

Runners:
- Windows: `windows-latest`, Visual Studio 17 2022 generator, MSVC
- macOS: `macos-latest`, Xcode generator

JUCE FetchContent is cached by hashing `CMakeLists.txt` to avoid re-downloading on
every run.

`FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: true` is set at the workflow env level to opt
into the Node 24 runtime for all actions (GitHub is deprecating Node 20 runners).

### Artifact structure gotcha

`upload-artifact@v4` uses the **specified path as the root** of the zip — it does not
include the directory named in the path itself. For example:

```yaml
path: "build/RC20Clone_artefacts/Release/VST3/RC-20 Clone.vst3"
# artifact contains: Contents/Info.plist, Contents/MacOS/..., etc.
# bundle directory name is LOST
```

```yaml
path: "build/RC20Clone_artefacts/Release/VST3"
# artifact contains: RC-20 Clone.vst3/Contents/..., etc.
# bundle directory name is PRESERVED ✓
```

The workflow uploads the **parent directory** (`VST3/` and `AU/`) for this reason.
The install scripts depend on this structure.

## Install scripts

`scripts/install-macos.sh` and `scripts/install-windows.ps1` automate the
fetch → extract → copy workflow for end-users and developer testing.

Both require the **GitHub CLI** (`gh`) to be installed and authenticated (`gh auth login`).

They:
1. Derive the GitHub repo from `git remote get-url origin` (works on forks)
2. Find the latest successful `build.yml` run on `main`
3. Download artifacts via `gh run download` into separate subdirs (VST3 and AU
   must go into different dirs — both contain `Contents/` at the root and will
   collide if extracted to the same location)
4. Copy to the system plugin directory

macOS script also runs `xattr -rd com.apple.quarantine` on installed bundles so
Gatekeeper doesn't block unsigned binaries during development.

Windows script auto-elevates to Administrator (required to write to `Program Files`).

**macOS install locations:**
- VST3: `/Library/Audio/Plug-Ins/VST3/`
- AU:   `/Library/Audio/Plug-Ins/Components/`

**Windows install location:**
- VST3: `C:\Program Files\Common Files\VST3\`

## JUCE 8 API gotchas (already bitten us once each)

- **`std::atomic<float>*` dereferencing.** `*atomicPtr` returns an `atomic<float>`,
  not a `float`. Passing that directly into a template function (e.g.
  `Decibels::decibelsToGain`) instantiates the template on `atomic<float>`, which
  tries to copy-construct an atomic and fails. Always call `.load()` first:
  ```cpp
  const float db = outputLevelParam_->load();
  ```

- **`juce::Font` constructor deprecated.** `Font(float size, int style)` is gone.
  Use `FontOptions`:
  ```cpp
  label.setFont(juce::FontOptions(14.0f, juce::Font::bold));
  ```

- **`AlertWindow::showInputBoxAsync` removed.** Use `FileChooser::launchAsync`
  instead. The `FileChooser` instance must be kept alive as a member — if it goes
  out of scope the callback is never called.

- **Initializer-list `addAndMakeVisible` loops don't compile** when the list mixes
  component subclasses (type deduction fails). Call `addAndMakeVisible` individually.

- **Constructor parameter `processor` shadows `AudioProcessorEditor::processor`.**
  Clang's `-Wshadow-field` treats this as an error under
  `juce_recommended_warning_flags`. Rename the constructor param (e.g. `p`).

## Platform notes

- The Linux build (`.so`) cannot be loaded in a Windows DAW. CI produces the
  Windows binary; do not try to cross-compile manually.
- **Windows builds:** use the GitHub Actions artifact or build natively with
  Visual Studio 2022 (Desktop C++ workload). CMakeLists.txt is cross-platform.
- **macOS code signing:** AU and VST3 built in CI are unsigned. End-users need to
  strip the quarantine flag (`xattr -rd com.apple.quarantine <bundle>`) or the
  install script does it automatically. A Developer ID certificate is required
  before public distribution.
- **pluginval:** The prebuilt Linux binary requires glibc 2.35+. On this machine
  (glibc 2.31) it must be built from source. Once built, the binary is at
  `/tmp/pluginval/build/pluginval_artefacts/Release/pluginval`.

## Architecture decisions (the "why" behind the code)

These were settled before writing Phase 1 and should not be revisited without
good reason.

- **Fixed signal chain, not dynamic.** Modules are stored in a
  `std::array<std::unique_ptr<EffectModule>, 6>` in processing order:
  Noise → Wobble → Distortion → Space → Magic → Limit.
  Reorderable chains are a future option, not a current goal.

- **Single `DriftGenerator` in the processor.** It runs once per block and passes
  a single `float driftValue` to all modules via `ProcessContext`. Modules do not
  own generators.

- **Modes are mutually exclusive** (`AudioParameterChoice`, not bool flags).
  Each mode will be a `ProcessingMode` subclass (strategy pattern). Crossfading
  between modes on change lives in the module, not the mode.

- **`juce::dsp::AudioBlock` in `ProcessContext`.** In-place, zero-copy. Do not
  wrap in `ProcessContextReplacing` — the custom `ProcessContext` struct is
  intentional to carry the drift value alongside the audio.

- **Parameter IDs are all `snake_case`**, centralised in `ParameterIDs.h`. Never
  hard-code an ID string anywhere else.

- **`AudioParameterFloat` uses `juce::ParameterID{id, 1}`** (with version hint
  `1`), not the deprecated string-only constructor.

- **`SmoothedValue` ramps:** amount knobs use 20 ms, bypass crossfade uses 10 ms.
  These are initialised in each module's `prepare()` call.

## Known issues

- **pluginval locale test fails on Linux (strictness 5).** JUCE's `FTTypefaceList`
  calls `setlocale(LC_ALL, "C")` internally during font initialisation. This
  permanently changes the process locale before our constructor body runs, so
  there is no reliable way to restore the host's original locale from within the
  plugin. This is a well-known JUCE/Linux issue, affects most JUCE plugins, and
  has no impact on Windows DAW behaviour or audio correctness. All other pluginval
  tests pass at strictness 5.

### Phase 2 implementation order

Implement modules in this order — each builds on the complexity of the last:

1. **NoiseModule** — no time-domain state, no oversampling; good warm-up
   - TapeHiss: IIR-shaped white noise (pink filter)
   - VinylCrackle: Poisson-distributed random impulses
   - ElectricalHum: 50/60 Hz sine + harmonics

2. **MagicModule** — also stateless per-sample
   - BitCrush: `floor(x * steps + 0.5) / steps` where steps = 2^bits
   - Decimate: hold every Nth sample (sample-rate reduction)
   - Drift modulates bit depth / hold rate

3. **LimitModule** — output stage; useful to have working early
   - Envelope follower (attack/release) + soft-knee gain reduction
   - `tanh` saturation at higher amounts

4. **DistortionModule** — introduces oversampling
   - Use `juce::dsp::Oversampling` (2× or 4×)
   - Tape (soft clip), Tube (asymmetric waveshaper), Transistor (hard clip)

5. **WobbleModule** — delay line DSP; `ModulatedDelayLine` stubs already in place
   - LFO modulates delay time via cubic Hermite interpolation
   - Stereo width from L/R LFO phase offset

6. **SpaceModule** — most open-ended; `sizeParam_` and `toneParam_` already wired
   - `juce::dsp::Reverb` (Freeverb) is a fast start; FDN reverb is the upgrade path
   - Delay mode: feedback delay line with `toneParam_` controlling damping

Also implement `DriftGenerator` properly (multi-LFO + band-limited noise) before or
alongside WobbleModule — earlier modules can use a simple sine placeholder.

## Audio thread rules

Strictly enforced — do not add exceptions:

- No heap allocation in `processBlock` or any function it calls
- No mutex locks on the audio thread
- Parameter values are read via cached `std::atomic<float>*` pointers only
- `prepareParameters()` is message-thread only (called at construction)
- `prepare()` and `process()` are audio-thread only
